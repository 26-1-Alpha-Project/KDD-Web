# 9~15주차 활동 보고서 (프론트엔드)

## 1. 중간 평가 피드백 분석 및 고도화 계획 수립 (9주차)

중간 평가에서 받은 피드백을 항목별로 분류하고 대응 우선순위를 정했다. UI 완성도와 사용자 흐름에 대한 긍정적 평가를 확인하는 한편, 실제 API 연동 상태와 에러 핸들링 부족이 지적되어 이를 4차 스프린트의 핵심 과제로 설정했다. PM 역할로 고도화 단계의 스프린트 계획을 수립하여, 9주차에는 사용자 편의 기능 보강, 10~11주차에는 관리자 기능 개발, 12주차 이후에는 배포·테스트·안정화로 로드맵을 확정했다. 백엔드 팀과 주간 싱크 미팅을 정례화하여 API 명세 변경 사항을 실시간으로 공유하는 체계도 마련했다.

> **사진 추천**: 중간 평가 피드백 정리 시트, 4차 스프린트 로드맵 타임라인

## 2. FAQ 바로가기 및 답변 피드백 기능 구현 (9주차)

FAQ 바로가기 버튼을 채팅 웰컴 화면의 추천 질문 영역과 사이드바 하단에 배치하여, 사용자가 채팅 중에도 자주 묻는 질문 목록에 빠르게 접근할 수 있도록 했다. SuggestedQuestions 컴포넌트에 FAQ 인기 항목을 동적으로 노출하는 로직을 추가하고, 클릭 시 해당 FAQ 답변으로 바로 이동하는 라우팅을 연결했다. 답변 피드백 컴포넌트를 개발하여 FAQItem과 ChatMessage 양쪽에 통합했으며, useFAQ 훅에서 피드백 상태를 토글 방식으로 관리하도록 구현했다. POST /faqs/{faqId}/feedback API 연동 구조도 함께 설계하여 백엔드 준비 완료 시 즉시 연결 가능한 상태를 확보했다.

> **사진 추천**: 채팅 웰컴 화면에 FAQ 바로가기 버튼이 배치된 모습, FAQ 답변 하단의 좋아요/싫어요 피드백 버튼 UI

## 3. 코드 리팩토링 및 컴포넌트 구조 개선 (9주차)

채팅·FAQ·자료 페이지에서 반복되던 검색 필터링, 페이지네이션, 정렬 로직을 useDocuments, useFAQ, useAdminDocuments 등 도메인별 커스텀 훅으로 분리하여 컴포넌트 간 중복을 제거했다. 각 훅은 상태 관리, 필터 조건 변경, 정렬 전환, 페이지 리셋 등을 캡슐화하여 페이지 컴포넌트에서는 훅의 반환값만으로 UI를 렌더링할 수 있도록 했다. 컴포넌트 폴더 구조도 재정비하여 chat/, faq/, resources/, admin/, sidebar/, tour/ 등 도메인별로 명확히 분리하고, 공통 UI 요소는 ui/ 폴더로 통합했다. 타입 정의도 types/api/ 하위에 chat, faq, document, admin, common으로 세분화하여 API 명세와 1:1 대응되도록 정리했다.

> **사진 추천**: 리팩토링 전후 폴더 구조 비교, 커스텀 훅 분리 다이어그램

## 4. 관리자 대시보드 레이아웃 및 통계 카드 구현 (10주차)

백엔드 팀과 관리자 API 인터페이스를 확정한 뒤 관리자 페이지 개발에 본격 착수했다. 관리자 전용 레이아웃을 구성하여 좌측 네비게이션에 대시보드, 문서 관리, FAQ 관리 메뉴를 배치하고, 각 메뉴 클릭 시 해당 탭으로 전환되는 구조를 잡았다. 대시보드 메인 화면에는 StatCard 컴포넌트를 개발하여 총 질문 수, 평균 신뢰도, 활성 사용자 수, 등록 문서 수 등 핵심 지표를 카드 형태로 배치했다. StatsTab에서는 useAdminStats 훅을 통해 통계 데이터를 로드하고, 사용자 유형별·학과별·학년별 분포를 시각적으로 표현하는 구조를 완성했다.

> **사진 추천**: 관리자 대시보드에 통계 카드 4개가 나란히 배치된 화면, 사용자 유형별 분포 차트

## 5. 관리자 로그인 및 권한 분기 처리 (10주차)

관리자 로그인은 기존 Google OAuth 인증 플로우에 역할 기반 분기를 추가하여 처리했다. 로그인 후 반환되는 사용자 정보에서 role 필드를 확인하고, admin 역할인 경우 /admin 경로로 리다이렉트하는 로직을 구현했다. proxy.ts에서 /admin 하위 경로 접근 시 인증 토큰과 역할을 검증하여, 권한이 없는 사용자는 메인 페이지로 리다이렉트되도록 설정했다. 관리자 페이지 내에서도 각 탭별로 접근 권한을 세분화할 수 있는 구조를 미리 설계해두어, 향후 권한 체계 확장에 대비했다.

> **사진 추천**: 관리자 로그인 후 대시보드로 리다이렉트되는 흐름, 권한 검증 로직 코드 스니펫

## 6. 문서 업로드 및 관리 기능 구현 (11주차)

관리자 문서 관리 탭에서 DocumentUpload 컴포넌트를 통해 PDF 파일을 업로드하는 기능을 구현했다. 드래그앤드롭 영역과 파일 선택 버튼을 제공하고, 업로드 진행 시 프로그레스 바를 표시하며, 완료 후 파일명·카테고리·상태가 DocumentTable에 즉시 반영되도록 했다. useAdminDocuments 훅에서 문서 검색, 삭제, 재처리 기능을 구현하여, 임베딩 실패 상태의 문서를 관리자가 재시도할 수 있는 워크플로우를 완성했다. 문서 상태는 processing, completed, failed, reprocessing 네 가지로 관리하며, 상태별로 뱃지 색상을 다르게 표시하여 현황을 직관적으로 파악할 수 있도록 했다.

> **사진 추천**: 문서 업로드 드래그앤드롭 영역, 문서 테이블에서 상태별 뱃지가 표시된 화면

## 7. FAQ 후보 관리 및 백엔드 API 연결 (11주차)

FAQ 관리 탭에서 등록된 FAQ 목록과 AI가 자동 생성한 FAQ 후보 목록을 탭으로 분리했다. 등록 FAQ에는 질문, 답변 미리보기, 토픽, 작성일, 피드백 수를 표시하고 삭제 기능을 제공했다. 후보 목록에서는 각 항목에 질문, 초안 답변, 출현 빈도, 토픽을 노출하고, 승인/반려 버튼을 배치하여 useAdminFAQ 훅의 approveCandidate, rejectCandidate 함수로 상태를 전환하도록 구현했다. 승인 시 topic을 지정할 수 있도록 요청 body에 topic 필드를 포함하는 구조를 설계했다.

동시에 FAQ 조회 및 관리자 FAQ CRUD를 실제 백엔드 API와 연결하는 작업을 진행했다. 공개 FAQ 엔드포인트와 관리자 FAQ 엔드포인트를 호출하도록 서비스 레이어를 교체하고, 백엔드 페이지 인덱스가 0-base인 반면 프론트는 1-base를 사용하는 차이를 변환 로직으로 흡수했다. 응답 shape 차이를 처리하기 위한 매퍼 함수를 추가하여 타입 안정성을 확보했다.

> **사진 추천**: FAQ 후보 목록에서 승인/반려 버튼이 있는 화면, FAQ 관리 탭 등록/후보 분리 UI

## 8. 프로덕션 배포 및 런타임 안정화 (12주차)

Vercel을 통해 프로덕션 환경에 배포하고 환경 변수를 개발·프로덕션으로 분리하여 독립 관리되도록 설정했다. API 엔드포인트, OAuth 클라이언트 ID 등 민감 설정을 Vercel 환경 변수로 이관하고, next.config.ts의 rewrites 규칙이 프로덕션에서도 정상 동작하는지 검증했다. PDFViewer에 dynamic import를 적용하여 SSR 환경에서 DOMMatrix 참조 에러가 발생하던 문제를 해결하고, react-pdf 워커 번들이 프로덕션 빌드에서 정상 분리되는지 확인했다.

백엔드에서 아직 구현되지 않은 엔드포인트 11개를 mock-only로 통일 처리하고, NEXT_PUBLIC_USE_MOCK 환경변수 기반으로 mock과 실제 API 호출을 분기하는 구조를 정리하여 프로덕션 환경에서 mock 코드가 불필요하게 포함되지 않도록 했다. 16차 회의에서 백엔드 팀이 요청한 관리자용 사용자 프로필 삭제 버튼과 확인 모달 UI도 개발했다.

> **사진 추천**: Vercel 배포 대시보드 화면, 프로덕션 사이트 로그인 화면, 관리자 프로필 삭제 확인 모달

## 9. 사용자별 토큰 조회 페이지 및 프로덕션 검증 (12주차)

16차 회의에서 백엔드 팀이 요청한 사용자별 채팅 토큰 사용량 조회 페이지의 레이아웃과 API 연동 구조를 설계했다. 관리자가 각 사용자의 토큰 소비 현황을 확인할 수 있도록 테이블 형태의 UI를 구성하고, 향후 백엔드 API가 완성되면 즉시 연결할 수 있는 서비스 함수 구조를 준비했다.

프로덕션 사이트에서 전체 기능 동작을 검증했다. 로그인부터 채팅 질문, 출처 열람, FAQ 탐색, 자료 검색, 관리자 페이지 접근까지 주요 사용자 흐름을 E2E로 확인하고, 프로덕션 환경 특유의 이슈가 없는지 점검했다. SSE 스트리밍이 프로덕션 도메인에서도 안정적으로 동작하는지, 인증 토큰 갱신 플로우가 정상인지 등을 중점적으로 확인했다.

> **사진 추천**: 사용자별 토큰 조회 페이지 레이아웃, 프로덕션 환경 채팅 동작 화면

## 10. 오픈 베타 테스트 및 인라인 Citation 구현 (13주차)

교내 학생을 대상으로 오픈 베타 테스트를 시작했다. 주요 시나리오 가이드를 공유하여 질문 입력부터 답변 확인, 출처 열람, FAQ 탐색까지의 흐름을 테스터들이 자연스럽게 경험할 수 있도록 안내했다. 베타 운영 중 SSE 진단 로그를 활용하여 스트리밍 연결 안정성을 실시간으로 모니터링하고, 연결 끊김이나 파싱 실패가 발생하는 패턴을 추적했다.

AI 서버에서 답변 본문에 삽입하는 Citation 마커를 프론트엔드에서 파싱하여 인라인 출처 표시로 렌더링하는 기능을 구현했다. parseCitations 유틸리티를 개발하여 본문 내 출처 마커를 인식하고 텍스트와 출처 참조를 분리하는 로직을 작성했다. AnswerWithCitations 컴포넌트를 신규 개발하여 구글 AI Overview 스타일의 인라인 출처 뱃지를 렌더링하고, 동일 출처가 여러 번 등장할 때 번호를 그룹화하며, 사용자가 답변을 복사할 때 마커가 자동으로 제거되도록 처리했다. 같은 문서에서 다중 chunk가 인용될 때 번호 중복 제거가 정상 동작하는지 검증하고, 출처 dedup 로직이 실제 백엔드 응답에서도 정상 작동하는 것을 확인했다.

> **사진 추천**: 베타 테스트 안내 메시지, 채팅 답변에 인라인 Citation 뱃지가 표시된 화면

## 11. 모바일 뷰 점검 및 베타 안정성 확인 (13주차)

Chrome DevTools 반응형 모드를 활용하여 전체 페이지의 모바일 뷰 레이아웃을 점검했다. 채팅 페이지에서 입력창 위치와 터치 타겟 크기가 적절한지, FAQ 페이지의 토픽 트리가 좁은 화면에서도 정상 표시되는지, 자료 페이지의 카테고리 트리와 문서 목록이 모바일에서 깨지지 않는지 확인했다. PDF 뷰어의 모바일 대응 상태도 점검하여 페이지 네비게이션 버튼이 터치로 조작 가능한지 확인했다.

베타 운영 중 발견된 채팅 UX 잔여 이슈를 점검했다. 이전에 수정한 입력 잔류 버그와 연타 버그가 재발하지 않는지 확인하고, 전반적인 서비스 안정성을 모니터링했다. 베타 기간 동안 수집된 잔여 이슈를 백로그로 정리했다.

> **사진 추천**: Chrome DevTools 반응형 모드에서 채팅 페이지 모바일 뷰, PDF 뷰어 모바일 화면

## 12. 시연 준비 및 코드 정리 (14주차)

최종 시연 시나리오를 확정했다. 신입생이 장학금에 대해 질문하면 AI가 Citation 포함 답변을 생성하고, 출처 문서를 열람한 뒤 FAQ에서 관련 질문을 탐색하며, 관리자가 통계를 확인하는 전체 흐름으로 구성했다. 시연에 사용할 데이터를 정비하여 채팅 세션, FAQ 항목, 문서 카테고리 등이 데모에 적합한 상태가 되도록 세팅했다.

프론트엔드 소스코드에서 미사용 import, TODO 주석, console.log를 일괄 정리하고, 주요 컴포넌트에 JSDoc 주석을 보강하여 코드 가독성을 개선했다. ChatMessage, AnswerWithCitations, PDFViewer, AuthProvider 등 핵심 모듈의 역할과 props를 명확히 문서화했다.

> **사진 추천**: 시연 시나리오 흐름도, JSDoc 주석이 보강된 코드 스니펫

## 13. 최종 보고서 및 발표 자료 제작 (14주차)

최종 결과 보고서의 프론트엔드 파트를 작성했다. Next.js App Router 기반 아키텍처, 컴포넌트 설계 원칙, SSE 스트리밍 구현 방식, 커스텀 훅 기반 상태 관리 전략 등 기술적 의사결정 과정을 상세히 기술했다. 발표용 시스템 아키텍처 다이어그램과 주요 화면 스크린샷을 정리하고, 발표 자료 PPT의 프론트엔드 섹션을 작성했다.

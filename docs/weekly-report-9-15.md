# 9~15주차 활동 보고서 (프론트엔드)

## 1. 중간 평가 피드백 분석 및 고도화 계획 수립 (9주차)

중간 평가에서 받은 피드백을 항목별로 분류하고 대응 우선순위를 정했다. UI 완성도와 사용자 흐름에 대한 긍정적 평가를 확인하는 한편, 실제 API 연동 상태와 에러 핸들링 부족이 지적되어 이를 4차 스프린트의 핵심 과제로 설정했다. PM 역할로 고도화 단계의 스프린트 계획을 수립하여, 9주차에는 사용자 편의 기능 보강, 10~11주차에는 관리자 기능 개발, 12주차 이후에는 배포·테스트·안정화로 로드맵을 확정했다. 백엔드 팀과 주간 싱크 미팅을 정례화하여 API 명세 변경 사항을 실시간으로 공유하는 체계도 마련했다.

> **사진 추천**: 중간 평가 피드백 정리 시트, 4차 스프린트 로드맵 타임라인

## 2. FAQ 바로가기 및 답변 피드백 기능 구현 (9주차)

FAQ 바로가기 버튼을 채팅 웰컴 화면(ChatWelcome)의 추천 질문 영역과 사이드바 하단에 배치하여, 사용자가 채팅 중에도 자주 묻는 질문 목록에 빠르게 접근할 수 있도록 했다. SuggestedQuestions 컴포넌트에 FAQ 인기 항목을 동적으로 노출하는 로직을 추가하고, 클릭 시 해당 FAQ 답변으로 바로 이동하는 라우팅을 연결했다. 답변 피드백(좋아요/싫어요) 컴포넌트를 개발하여 FAQItem과 ChatMessage 양쪽에 통합했으며, useFAQ 훅에서 피드백 상태를 토글 방식으로 관리하도록 구현했다. POST /faqs/{faqId}/feedback API 연동 구조도 함께 설계하여 백엔드 준비 완료 시 즉시 연결 가능한 상태를 확보했다.

> **사진 추천**: 채팅 웰컴 화면에 FAQ 바로가기 버튼이 배치된 모습, FAQ 답변 하단의 좋아요/싫어요 피드백 버튼 UI

## 3. 코드 리팩토링 및 컴포넌트 구조 개선 (9주차)

채팅·FAQ·자료 페이지에서 반복되던 검색 필터링, 페이지네이션, 정렬 로직을 useDocuments, useFAQ, useAdminDocuments 등 도메인별 커스텀 훅으로 분리하여 컴포넌트 간 중복을 제거했다. 각 훅은 상태 관리, 필터 조건 변경, 정렬 전환, 페이지 리셋 등을 캡슐화하여 페이지 컴포넌트에서는 훅의 반환값만으로 UI를 렌더링할 수 있도록 했다. 컴포넌트 폴더 구조도 재정비하여 chat/, faq/, resources/, admin/, sidebar/, tour/ 등 도메인별로 명확히 분리하고, 공통 UI 요소(Button, Input, Dialog, Table 등)는 ui/ 폴더로 통합했다. 타입 정의도 types/api/ 하위에 chat, faq, document, admin, common으로 세분화하여 API 명세와 1:1 대응되도록 정리했다.

> **사진 추천**: 리팩토링 전후 폴더 구조 비교, 커스텀 훅 분리 다이어그램

## 4. 관리자 대시보드 레이아웃 및 통계 카드 구현 (10주차)

백엔드 팀과 관리자 API 인터페이스(GET /admin/statistics, GET /admin/documents 등)를 확정한 뒤 관리자 페이지 개발에 본격 착수했다. 관리자 전용 레이아웃(admin/layout.tsx)을 구성하여 좌측 네비게이션에 대시보드, 문서 관리, FAQ 관리 메뉴를 배치하고, 각 메뉴 클릭 시 해당 탭으로 전환되는 구조를 잡았다. 대시보드 메인 화면에는 StatCard 컴포넌트를 개발하여 총 질문 수, 평균 신뢰도, 활성 사용자 수, 등록 문서 수 등 핵심 지표를 카드 형태로 배치했다. StatsTab에서는 useAdminStats 훅을 통해 통계 데이터를 로드하고, 사용자 유형별(학생/교직원)·학과별·학년별 분포를 시각적으로 표현하는 구조를 완성했다.

> **사진 추천**: 관리자 대시보드에 통계 카드 4개가 나란히 배치된 화면, 사용자 유형별 분포 차트

## 5. 관리자 로그인 및 권한 분기 처리 (10주차)

관리자 로그인은 기존 Google OAuth 인증 플로우에 역할(role) 기반 분기를 추가하여 처리했다. 로그인 후 반환되는 사용자 정보에서 role 필드를 확인하고, admin 역할인 경우 /admin 경로로 리다이렉트하는 로직을 구현했다. middleware.ts에서 /admin 하위 경로 접근 시 인증 토큰과 역할을 검증하여, 권한이 없는 사용자는 메인 페이지로 리다이렉트되도록 설정했다. 관리자 페이지 내에서도 각 탭(문서 관리, FAQ 관리)별로 접근 권한을 세분화할 수 있는 구조를 미리 설계해두어, 향후 권한 체계 확장에 대비했다.

> **사진 추천**: 관리자 로그인 후 대시보드로 리다이렉트되는 흐름, middleware 권한 검증 로직 코드 스니펫

## 6. 문서 업로드 및 관리 기능 구현 (11주차)

관리자 문서 관리 탭(DocsTab)에서 DocumentUpload 컴포넌트를 통해 PDF 파일을 업로드하는 기능을 구현했다. 드래그앤드롭 영역과 파일 선택 버튼을 제공하고, 업로드 진행 시 프로그레스 바를 표시하며, 완료 후 파일명·카테고리·상태가 DocumentTable에 즉시 반영되도록 했다. useAdminDocuments 훅에서 문서 검색, 삭제, 재처리(reprocess) 기능을 구현하여, 임베딩 실패 상태의 문서를 관리자가 재시도할 수 있는 워크플로우를 완성했다. 문서 상태는 processing, completed, failed, reprocessing 네 가지로 관리하며, 상태별로 뱃지 색상을 다르게 표시하여 현황을 직관적으로 파악할 수 있도록 했다.

> **사진 추천**: 문서 업로드 드래그앤드롭 영역, 문서 테이블에서 상태별 뱃지가 표시된 화면

## 7. FAQ 후보 관리 및 신뢰도 차트 구현 (11주차)

FAQ 관리 탭(FAQTab)에서 등록된 FAQ 목록(FAQTable)과 AI가 자동 생성한 FAQ 후보 목록(FAQCandidateTable)을 탭으로 분리했다. 등록 FAQ에는 질문, 답변 미리보기, 토픽, 작성일, 피드백 수를 표시하고 삭제 기능을 제공했다. 후보 목록에서는 각 항목에 질문, 초안 답변, 출현 빈도, 토픽을 노출하고, 승인/반려 버튼을 배치하여 useAdminFAQ 훅의 approveCandidate, rejectCandidate 함수로 상태를 전환하도록 구현했다. 승인 시 topic을 지정할 수 있도록 POST /admin/faqs/candidates/{id}/approve 요청 body에 topic 필드를 포함하는 구조를 설계했다. 신뢰도별 답변 분포 시각화를 위한 차트 컴포넌트도 구현하여 관리자가 AI 답변 품질 현황을 한눈에 파악할 수 있도록 했다.

> **사진 추천**: FAQ 후보 목록에서 승인/반려 버튼이 있는 화면, 신뢰도 분포 차트

## 8. 프로덕션 배포 및 성능 최적화 (12주차)

Vercel을 통해 프로덕션 환경에 배포하고 커스텀 도메인을 연결했다. 배포 파이프라인에서 환경 변수(API 엔드포인트, OAuth 클라이언트 ID 등)를 분리하여 개발/스테이징/프로덕션 환경별로 독립 관리되도록 설정했다. Lighthouse로 성능 프로파일링을 수행하여 LCP 2.1초, CLS 0.05, INP 180ms 등 Core Web Vitals 지표를 측정하고, 개선 목표를 설정했다. 초기 번들 사이즈를 줄이기 위해 관리자 페이지와 PDF 뷰어에 dynamic import를 적용하고, 이미지 최적화에 Next.js의 Image 컴포넌트를 활용했다. 서버 컴포넌트와 클라이언트 컴포넌트 경계를 재정비하여 불필요한 클라이언트 번들이 포함되지 않도록 최적화했다.

> **사진 추천**: Lighthouse 성능 점수 리포트, Vercel 배포 대시보드 화면

## 9. SEO 설정 및 접근성 검토 (12주차)

SEO 메타 태그(title, description, canonical)와 OG 이미지를 각 페이지별로 설정하여 외부 공유 시 미리보기가 정상 표시되도록 했다. Next.js의 Metadata API를 활용하여 동적 라우트(/chat/[id], /resources/[fileId])에서도 적절한 메타 정보가 생성되도록 구현했다. 접근성(a11y) 검토를 병행하여 키보드 네비게이션으로 모든 인터랙티브 요소에 접근 가능한지 확인하고, aria-label 누락, 포커스 인디케이터 미표시, 색상 대비 부족 등 주요 항목을 점검하여 개선했다. 부하 테스트 시나리오를 설정하여 동시 접속 50명 상황에서 SSE 스트리밍 연결이 안정적으로 유지되는지도 검증했다.

> **사진 추천**: OG 이미지 미리보기가 표시된 카카오톡/슬랙 공유 화면, 접근성 검사 결과 리포트

## 10. 베타 테스트 진행 및 피드백 수집 (13주차)

베타 테스터 15명을 모집하여 1주간 사용성 테스트를 진행했다. 주요 시나리오(질문 입력→답변 확인→출처 열람→FAQ 탐색→자료 검색)별로 태스크 완료율과 소요 시간을 측정하고, 각 시나리오 종료 후 SUS(System Usability Scale) 설문을 수집했다. 수집된 피드백을 Critical/High/Medium/Low로 분류하여 우선순위별 수정 계획을 수립했다. 채팅 입력창의 전송 버튼 위치가 모바일에서 엄지 도달 범위 밖이라는 피드백을 반영하여 위치를 조정하고, 출처 카드의 클릭 영역이 좁다는 의견에 따라 터치 타겟을 48px 이상으로 확대했다. FAQ 카테고리 필터의 초기 선택 상태가 불명확하다는 지적에 따라 "전체" 탭에 활성 스타일을 강화했다.

> **사진 추천**: 베타 테스트 피드백 정리 시트(우선순위별 분류), SUS 점수 결과 차트

## 11. 크로스 브라우저 및 모바일 기기 테스트 (13주차)

크로스 브라우저 테스트를 Chrome, Safari, Firefox 최신 버전에서 수행했다. Safari에서 backdrop-filter 속성이 적용되지 않는 문제, Firefox에서 flex gap 계산이 미세하게 다른 문제 등 CSS 렌더링 차이를 발견하고 수정했다. 이벤트 핸들링에서도 Safari의 passive 이벤트 리스너 기본값 차이로 인한 스크롤 이슈를 해결했다. 모바일 기기 테스트에서는 iOS Safari의 가상 키보드 등장 시 뷰포트 리사이즈로 채팅 입력창이 가려지는 문제를 visualViewport API로 대응하고, Android Chrome에서 주소창 높이 변동에 의한 레이아웃 깨짐을 dvh 단위 적용으로 해결했다. 터치 인터랙션에서 300ms 딜레이와 더블탭 줌 방지도 함께 처리했다.

> **사진 추천**: Chrome/Safari/Firefox 비교 화면, 모바일 기기(iPhone, Galaxy) 실물 테스트 사진

## 12. 최종 버그 수정 및 성능 튜닝 (14주차)

GitHub Issues에 등록된 잔여 버그를 모두 처리했다. 빈 검색 결과 시 안내 메시지 미표시, 네트워크 오류 시 무한 로딩, 세션 만료 후 API 호출 시 무응답 등 엣지 케이스에 대한 에러 핸들링을 보강했다. SSE 스트리밍 중 연결이 끊어졌을 때 자동 재연결을 시도하고, 3회 실패 시 사용자에게 재시도 버튼을 표시하는 폴백 로직도 추가했다. 성능 튜닝으로는 ChatMessageList에서 메시지가 많아질 때 발생하던 불필요한 리렌더링을 React.memo와 useCallback으로 방지하고, FAQ 목록의 필터링 연산을 useMemo로 캐싱하여 체감 성능을 개선했다. Lighthouse 재측정으로 LCP 1.4초, CLS 0.02로 개선된 것을 확인했다.

> **사진 추천**: Lighthouse 점수 개선 전후 비교, 에러 핸들링 UI (네트워크 오류 알림, 재시도 버튼)

## 13. 산출물 정리 및 발표 자료 제작 (14주차)

PM 역할로 API 명세서를 최종 정리하고, 사용자 매뉴얼 초안을 작성했다. 매뉴얼에는 회원가입/로그인, 채팅 질문, FAQ 탐색, 자료 검색, 관리자 기능 등 주요 기능별 사용법을 스크린샷과 함께 기술했다. 최종 시연 시나리오를 확정하여 "신입생이 장학금에 대해 질문→AI 답변 확인→출처 문서 열람→FAQ에서 관련 질문 탐색→관리자가 통계 확인" 흐름으로 구성했다. 발표 자료에 들어갈 시스템 아키텍처 다이어그램, 컴포넌트 구조도, 주요 화면 스크린샷을 정리하고, 프로젝트 포트폴리오용 목업 이미지도 제작했다. 소스코드 전반에 걸쳐 주석을 정리하고, 미사용 코드와 TODO 주석을 제거하는 클린업 작업을 수행했다.

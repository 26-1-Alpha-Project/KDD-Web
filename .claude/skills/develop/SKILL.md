# 멀티 스테이지 개발 파이프라인

단순히 코드를 바로 짜는 대신, 3단계 파이프라인으로 품질을 보장한다.

## Stage 1: Plan (탐색 + 설계)

구현 전에 반드시 수행:

1. **기존 코드 탐색** — 재사용 가능한 컴포넌트, 훅, 유틸, 타입을 먼저 찾는다
   - `components/` 디렉토리에 비슷한 컴포넌트가 있는지
   - `hooks/`에 관련 훅이 있는지
   - `lib/`에 재사용 가능한 유틸이 있는지
   - `types/`에 이미 정의된 타입이 있는지
2. **Figma URL이 주어진 경우** — 디자인 컨텍스트를 조회하여 시각 정보 확보
3. **구현 계획 작성** — 사용자에게 제시:
   - 생성/수정할 파일 목록
   - 재사용할 기존 코드 목록
   - 새로 만들어야 하는 것 목록
   - 예상 의존 관계

4. **사용자 승인 대기** — "이대로 진행할까요?" 확인 후 다음 단계

## Stage 2: Implement (구현)

승인된 계획에 따라 구현:

1. **types/ 먼저** — 필요한 타입/인터페이스부터 정의
2. **lib/ 다음** — 유틸리티, API 함수
3. **hooks/ 다음** — 커스텀 훅
4. **components/ 다음** — UI 컴포넌트
5. **app/ 마지막** — 페이지에 조합

이 순서는 `.claude/rules/architecture.md`의 의존 방향을 따른 것이다.

구현 중 주의:
- 계획에 없는 파일을 임의로 생성하지 않는다
- 기존 컴포넌트의 Props를 변경할 때는 사용처도 함께 수정한다
- 기존 유틸이 있으면 새로 만들지 않고 재사용한다

## Stage 3: Review (자체 검증)

구현 완료 후 자동 수행:

1. **lint 실행** — `npm run lint` 통과 확인
2. **변경사항 리뷰** — `git diff`로 전체 변경 사항 확인:
   - 의도하지 않은 파일 변경이 없는지
   - 삭제된 export/prop이 없는지
   - 하드코딩 색상이 없는지
   - `console.log`가 남아있지 않은지
3. **결과 보고** — 사용자에게 변경 요약 제시

## 단순 수정에는 사용하지 않는다

다음 경우에는 이 파이프라인 없이 바로 작업해도 된다:
- 텍스트/스타일 수정
- 버그 수정 (원인이 명확한 경우)
- 기존 컴포넌트에 작은 prop 추가
